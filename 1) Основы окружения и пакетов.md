### .env 

Переменные окружения (environment variables) — это динамические именованные значения, которые используются операционной системой или приложениями для хранения конфигурационных параметров.  
Они используются для хранения конфигурационных данных, которые могут изменяться в зависимости от среды выполнения (например, разработка, [тестирование](https://e.sfu-kras.ru/mod/lesson/view.php?id=1848964 "Тестирование"), production), без изменения исходного кода.

**Назначение**  
Хранение конфигурации: переменные окружения позволяют отделить конфигурацию от кода, что делает приложение более гибким и переносимым.  
[Безопасность](https://e.sfu-kras.ru/mod/lesson/view.php?id=1848965 "Безопасность"): чувствительные данные, такие как API-ключи, пароли и токены, не должны храниться в коде. Использование переменных окружения позволяет избежать их попадания в репозиторий.  
Управление средой выполнения: разные среды (development, staging, production) могут требовать различных настроек. Переменные окружения позволяют легко переключать конфигурацию без изменения кода.  
Упрощение развертывания: приложения могут быть развернуты в различных окружениях (локально, на сервере, в контейнере) с минимальными изменениями, если конфигурация вынесена в переменные окружения.

### pip/uv 
[UV](https://github.com/astral-sh/uv) ([сайт](https://astral.sh/)) — это чрезвычайно быстрый пакетный менеджер Python, написанный на Rust. Разработан как замена для pip и pip-tools. Помимо этого он может собой заменить venv и pyenv.

### Версионирование 

Обычный номер версии ДОЛЖЕН иметь формат X.Y.Z, где X, Y и Z — неотрицательные целые числа и НЕ ДОЛЖНЫ содержать ведущие нули. X — мажорная версия, Y — минорная версия и Z — патч-версия. Каждый элемент ДОЛЖЕН увеличиваться численно. Например: 1.9.0 ->1.10.0 -> 1.11.0.

1. Версия 1.0.0 определяет публичный API. После этого релиза номера версий увеличиваются в зависимости от того, как изменяется публичный API.

2. Патч-версия Z (x.y.Z | x > 0) ДОЛЖНА быть увеличена только если содержит обратно совместимые баг-фиксы. Определение баг-фикс означает внутренние изменения, которые исправляют некорректное поведение. 

3. Минорная версия (x.Y.z | x > 0) ДОЛЖНА быть увеличена, если в публичном API представлена новая обратно совместимая функциональность. Версия ДОЛЖНА быть увеличена, если какая-либо функциональность публичного API помечена как устаревшая (deprecated). Версия МОЖЕТ быть увеличена в случае реализации новой функциональности или существенного усовершенствования в приватном коде. Версия МОЖЕТ включать в себя изменения, характерные для патчей. Патч-версия ДОЛЖНА быть обнулена, когда увеличивается минорная версия.

4. Мажорная версия X (X.y.z | X > 0) ДОЛЖНА быть увеличена, если в публичном API представлены какие-либо обратно несовместимые изменения. Она МОЖЕТ включать в себя изменения, характерные для уровня минорных версий и патчей. Когда увеличивается мажорная версия, минорная и патч-версия ДОЛЖНЫ быть обнулены.

Точная версия: =1.2.3 или просто 1.2.3 Устанавливает именно указанную версию. 

Совместимая версия (Caret): ^1.2.3 Значение: Обновления с обратной совместимостью. Правила: ^1.2.3 → >=1.2.3 <2.0.0 
^0.2.3 → >=0.2.3 <0.3.0 (до 1.0.0 MINOR как MAJOR) 
^0.0.3 → >=0.0.3 <0.0.4 (только патчи) 

Приблизительная версия (Tilde): ~1.2.3 Значение: Только патчи или минорные обновления. Правила: ~1.2.3 → >=1.2.3 <1.3.0 
~1.2 → >=1.2.0 <1.3.0 
~1 → >=1.0.0 <2.0.0 
Совместимая версия Python (PEP 440): ~=1.2.3 Эквивалент: >=1.2.3, \==1.2.* Позволяет обновлять до любой версии, где все компоненты, кроме последнего указанного, остаются неизменными. 

Диапазон версий: >=1.2.0, <2.0.0 Гибкое задание границ. 
Например: >=1.2.0 <2.0.0 — от 1.2.0 включительно до 2.0.0 (исключая) 
1.2.0 || 1.3.0 — или 1.2.0, или 1.3.0
